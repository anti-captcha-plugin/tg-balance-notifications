# tg-balance-notifications

Bash-крон, который опрашивает один или несколько API-ключей Anti-Captcha, сравнивает баланс каждого с общей отсечкой и шлёт уведомление в Telegram, когда баланс ключа падает ниже неё.

Один экземпляр этого репо обслуживает одного клиента. У клиента может быть один или несколько API-ключей Anti-Captcha; все они используют общую отсечку и шлют в один и тот же Telegram-чат.

## Зависимости

- `bash` 4+
- `curl`
- `jq`
- Репо склонировано на хост, где будет крутиться cron

## Настройка Telegram-бота

### Создать бота

1. Откройте Telegram, начните диалог с [@BotFather](https://t.me/BotFather).
2. Отправьте `/newbot`.
3. Укажите отображаемое имя (любое), затем username, оканчивающийся на `bot` (например `myteam_balance_bot`).
4. BotFather пришлёт токен — строку вида `8037205173:AAGAn...` — это и есть `TELEGRAM_BOT_TOKEN`. Храните в секрете.

### Узнать ID чата

Алерты можно слать либо в личку бота, либо в группу. В обоих случаях самый простой путь — спросить у публичного бота, единственная работа которого — печатать ID.

**Личный чат**

1. В Telegram начните диалог с [@userinfobot](https://t.me/userinfobot) и отправьте `/start`.
2. Он сразу ответит вашим числовым user ID — это и есть `TELEGRAM_CHAT_ID`.
3. Также нажмите **Start** у вашего собственного бота (поиск по `@username`), иначе он не сможет вам писать.

**Групповой чат**

1. Добавьте вашего бота в группу (Группа → настройки группы → **Добавить участников** → поиск по `@username`). Права админа не нужны.
2. Добавьте в ту же группу [@getidsbot](https://t.me/getidsbot) — он сразу пишет `Chat ID: -100…` для этой группы.
3. Скопируйте это отрицательное число (**со знаком минус**) — это и есть ваш `TELEGRAM_CHAT_ID`.
4. Удалите `@getidsbot` из группы.

> **На заметку:** сторонние ID-боты видят сообщения группы, пока находятся в ней — добавляйте их только в чаты, контент которых не критичен, и удаляйте сразу после получения ID.
>
> **Подводный камень:** если базовая группа потом апгрейдится до супергруппы (часто происходит автоматически), chat ID меняется — обычно с маленького отрицательного числа на число, начинающееся с `-100`. Если алерты перестали приходить — заново возьмите ID.

## Конфигурация `.env`

`.env` лежит рядом со скриптами. Он **закоммичен** в репо с плейсхолдер-значениями; замените их на реальные.

### Поля

| Переменная | Назначение |
| --- | --- |
| `BALANCE_THRESHOLD` | Отсечка в долларах. Уведомление шлётся, если баланс ключа строго меньше этого значения. Общая для всех ключей. |
| `TELEGRAM_BOT_TOKEN` | Токен бота от BotFather, например `123456:ABC-DEF...`. |
| `TELEGRAM_CHAT_ID` | ID целевого чата (положительное целое для личного чата, отрицательное — обычно начинается с `-100` — для групп). |
| `KEY1`, `KEY2`, ... | Клиентские ключи Anti-Captcha. Скрипт итерирует `KEY1`, `KEY2`, ... и останавливается на первой незаданной (или закомментированной) переменной — поэтому нумерация должна идти подряд начиная с `KEY1`. |

### Пример — клиент с одним ключом

```env
BALANCE_THRESHOLD=2.00

TELEGRAM_BOT_TOKEN=123456:ABC-DEF...
TELEGRAM_CHAT_ID=-1001234567890

KEY1=11111111111111111111111111111111
# KEY2=22222222222222222222222222222222
# KEY3=33333333333333333333333333333333
# KEY4=44444444444444444444444444444444
```

Раскомментирован только `KEY1`, поэтому `check_balance.sh` опрашивает один ключ за прогон.

### Пример — клиент с четырьмя ключами

```env
BALANCE_THRESHOLD=2.00

TELEGRAM_BOT_TOKEN=123456:ABC-DEF...
TELEGRAM_CHAT_ID=-1001234567890

KEY1=11111111111111111111111111111111
KEY2=22222222222222222222222222222222
KEY3=33333333333333333333333333333333
KEY4=44444444444444444444444444444444
```

Раскомментированы все четыре ключа — скрипт обрабатывает их последовательно. Нужно больше четырёх? Просто добавьте `KEY5=...`, `KEY6=...`.

## Проверка и запуск

### Проверить конфигурацию

```sh
./validate_env.sh
```

Делает по порядку:

1. Все обязательные переменные заданы.
2. `BALANCE_THRESHOLD` парсится как положительное число.
3. `TELEGRAM_BOT_TOKEN` принимается методом `getMe`.
4. Шлёт тестовое сообщение в `TELEGRAM_CHAT_ID` (заодно проверяет, что бот может писать в этот чат).
5. Для каждого `KEY<N>`: вызывает Anti-Captcha `getBalance` и печатает баланс либо ошибку.

Каждая строка — `✅` или `❌`. Exit-код `0` только если все проверки прошли.

### Запустить проверку баланса

```sh
./check_balance.sh
```

При успехе молчит (рассчитан на cron). Шлёт в Telegram сообщение по каждому ключу только если:

- баланс ниже `BALANCE_THRESHOLD`, либо
- Anti-Captcha API вернул ошибку по этому ключу.

Завершается ненулевым кодом только при локальных проблемах (нет `.env`, нет нужной утилиты). Сетевые/API-сбои по отдельному ключу пишут warning в stderr и идут дальше — один проблемный ключ не убивает прогон.

### Поставить в cron

```cron
*/30 * * * * /path/to/tg-balance-notifications/check_balance.sh >> /var/log/tg-balance.log 2>&1
```

Подбирайте интервал под скорость расхода баланса у клиента — обычно подходит 15–60 минут.

## Формат Telegram-сообщения

Все сообщения в Telegram — на английском, независимо от языка README. Ключ показывается с вырезанной серединой: первые 6 символов, `...`, последние 4. Для 32-символьного ключа Anti-Captcha это раскрывает суммарно 10 символов — достаточно для идентификации без утечки.

Низкий баланс:

```
⚠️ Anti-Captcha low balance

Key: 5dc7c0...f516
Balance: $0.42
Threshold: $2.00
```

Ошибка API:

```
❌ Anti-Captcha balance check failed

Key: 5dc7c0...f516
Code: ERROR_KEY_DOES_NOT_EXIST
Description: Account authentication key doesn't exist
```

Тестовое сообщение от `validate_env.sh`:

```
✅ tg-balance-notifications: test message from validate_env.sh (server-01 2026-05-23T01:15:42Z)
```

Одно сообщение на ключ — удобно читать и сразу видно, какой ключ сработал.
